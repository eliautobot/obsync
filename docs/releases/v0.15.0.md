# Obsync v0.15.0

Obsync v0.15.0 makes paired computers recoverable without deleting their identity or losing their Obsync ledger, and fixes the Windows Desktop setup failure that could report a connection before installation actually completed.

## Reconnect without disconnecting

An offline computer now has a **Reconnect** action on its Sources card. The repair flow keeps the same computer ID and preserves:

- Watched folders and their Start/Pause/Stop state
- Indexed document records and comparison history
- Active desktop-vault assignment and cached vault knowledge
- Queued commands and generated Obsidian notes

The first option opens the already installed Desktop so a valid saved pairing can use **Start this PC**. If the local credential is missing or unauthorized, an administrator can generate a short-lived reconnect code and paste it into the latest official Desktop installer. The code rotates that computer's agent credential in place instead of creating a duplicate device.

## Honest connection status

Creating or consuming a pairing code no longer marks a computer online. The Sources page and pairing window wait until the actual background Desktop successfully authenticates and sends a heartbeat. If local installation fails, the dashboard stays offline and clearly indicates that repair is still required.

## Windows permission fix

The previous installer could save a new pairing and then try to overwrite the installed `Obsync-Desktop.exe` while Windows still had that file open. Windows rejected the write with `[Errno 13] Permission denied`, leaving a server record that looked connected for up to 90 seconds even though the background app was not running.

v0.15.0 now:

1. Stages the local executable before consuming a one-time code.
2. Detects a byte-identical installed executable and never overwrites it.
3. Stops the background scheduled task before replacing a genuinely changed locked executable.
4. Uses an atomic replacement with bounded lock retries and a clear recovery message.
5. Preserves the previous local vault and watched-folder configuration during same-server repair.

## Automatic recovery after a restart

Obsync Desktop no longer exits when its first connection attempt times out. This commonly occurred when Windows signed in before Docker Desktop, a VPN, or the network was ready. The background process remains alive and retries automatically. An unauthorized credential still stops safely and can be repaired through **Reconnect**.

**Start this PC** also starts an existing per-user scheduled task without administrator access. Elevation is required only if the task or installation itself must be repaired.

## Verification coverage

The release adds regression and stress tests for:

- Thirty consecutive in-place credential rotations with exactly one retained computer
- Watched-folder, document-ledger, vault, and local watcher preservation
- Old-token revocation and new-token authentication
- Pending/offline registration until a real heartbeat arrives
- Identical executable repair without any overwrite attempt
- A simulated Windows `Errno 13` lock followed by safe task stop and retry
- Installation failure before any pairing code is consumed
- A Desktop runtime that survives an initial server connection timeout
- Reconnect UI, expiry, polling, and error behavior

The server and Windows Desktop should both be updated to v0.15.0 for the complete behavior.
